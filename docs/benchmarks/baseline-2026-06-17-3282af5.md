# Benchmark baseline — 2026-06-17 @ `3282af5`

Point-in-time reference capture so future autobench/perf work has something to
diff against. Two halves:

1. **Competitor comparison** — YCSB, UltimaDB vs RocksDB / Fjall / ReDB.
2. **Autobench microbench snapshot** — the first-party journal / SMR-apply /
   multiwriter-commit metrics, recorded *non-destructively* alongside the
   committed gate baselines.

## ⚠️ Host & noise caveat — read first

These numbers were taken on a **noisy virtualized sandbox host** (4 cores), the
same class of machine whose run-to-run variance the committed
`autobench/baselines/*.json` describe as **up to ~2×**. Criterion confidence
intervals below show it directly (e.g. `3.4±0.98ms` on workload A/Fjall — ±29%,
and the strict tier's `554±75 ms` fsync-bound writes).

Consequences:

- **Absolute numbers are not portable.** Do not compare these against a run on
  a different machine.
- **Only same-machine / same-class relative tracking is meaningful.** Re-run the
  exact commands below on a comparable host to track UltimaDB's movement vs this
  point.
- For a trustworthy absolute baseline, re-run on the real bench machine and
  re-capture.

## 1. YCSB — competitor comparison (on-disk, two durability tiers)

**Config:** `NUM_RECORDS=10_000`, `OPS_PER_ITER=1_000`, `FIELD_SIZE=100 B`,
Zipfian θ=0.99. Criterion "burst" benches, full sampling (50 samples/bench).
All engines on **real disk** (`ULTIMA_BENCH_DIR`, ext4 `/dev/sda1`) and **commit
one transaction per operation** — so commit granularity is identical and the
only variable that moves between the two tiers below is the durability level.

> **⚠️ Two methodology traps this capture corrects** (an earlier single-table
> run got both wrong and produced a misleading "RocksDB wins writes" result):
> 1. **tmpfs:** `/tmp` here is a **tmpfs (RAM)**, so `tempfile::tempdir()` made
>    every "on-disk" engine in-memory with free fsyncs. Fixed via
>    `bench_disk_dir()` → set `ULTIMA_BENCH_DIR` to a real disk path.
> 2. **asymmetric write paths:** RocksDB ran with its **WAL disabled** (zero
>    disk-path work) while UltimaDB wrote its WAL; and ReDB **batched** all ~500
>    writes of a burst into one commit while the others committed per-op. Both
>    are now aligned (RocksDB WAL on; ReDB per-op).

Tier is selected by `ULTIMA_BENCH_DURABILITY` (`nondurable` default / `strict`).

### Tier 1 — non-durable (WAL/journal written, no fsync)

Every engine writes its WAL/journal to the OS page cache but **no commit blocks
on fsync** — measures the write path + access path, not disk-sync latency:

- **UltimaDB** — `Standalone { Durability::Eventual, WalWrite::Coalesced }`
  (WAL + bincode, async fsync).
- **RocksDB** — WAL **on**, `WriteOptions` default (`sync=false`), large memtable.
- **ReDB** — `Durability::None`, one write tx per op.
- **Fjall** — default `PersistMode::Buffer`.

(median ± from critcmp; **bold = fastest engine** in row):

| Workload | UltimaDB | Fjall | ReDB | RocksDB |
|---|---|---|---|---|
| **A** update-heavy (50/50 r/w) | **2.6 ms · 378 K/s** | 3.4 ms · 285 K/s | 13.7 ms · 71 K/s | 3.8 ms · 260 K/s |
| **B** read-mostly (95/5)       | **338 µs · 2.8 M/s** | 968 µs · 1009 K/s | 2.1 ms · 454 K/s | 1273 µs · 767 K/s |
| **C** read-only                | **136 µs · 7.0 M/s** | 667 µs · 1465 K/s | 734 µs · 1331 K/s | 927 µs · 1054 K/s |
| **D** read-latest              | **482 µs · 2026 K/s** | 1460 µs · 669 K/s | 2.6 ms · 381 K/s | 3.8 ms · 260 K/s |
| **E** short-ranges (scan)      | **997 µs · 980 K/s** | 19.1 ms · 51 K/s | 8.5 ms · 114 K/s | 437.5 ms · 2.2 K/s |
| **F** read-modify-write        | **2.9 ms · 338 K/s** | 3.6 ms · 273 K/s | 17.3 ms · 56 K/s | 4.1 ms · 238 K/s |

- **UltimaDB now leads all six** — including the write-heavy A/F that the broken
  earlier run handed to RocksDB. With RocksDB's WAL actually on and ReDB
  committing per-op, UltimaDB's `Arc<R>` no-serialization write path and in-memory
  B-tree win across the board.
- ReDB drops sharply on write/RMW mixes (A/F, 56–71 K/s): per-op `redb`
  transactions each do B-tree page bookkeeping — the per-op honesty tax that the
  earlier batched-commit version hid.
- RocksDB workload E (`~0.44 s`, 439×) remains a pathological scan path; high
  variance, directional only.

### Tier 2 — strict (fsync on every commit)

Every engine forces the per-op commit durable to disk — measures fsync-bound
durable-write throughput:

- **UltimaDB** — `Durability::Consistent` (commit blocks until WAL fsync).
- **RocksDB** — `WriteOptions::set_sync(true)`.
- **ReDB** — `Durability::Immediate`.
- **Fjall** — `persist(PersistMode::SyncAll)` after each write.

| Workload | UltimaDB | Fjall | ReDB | RocksDB |
|---|---|---|---|---|
| **A** update-heavy (50/50 r/w) | 554 ms · 1803/s | 477 ms · 2.0 K/s | **275 ms · 3.6 K/s** | 514 ms · 1947/s |
| **B** read-mostly (95/5)       | 56.7 ms · 17.2 K/s | 42.8 ms · 22.8 K/s | **32.9 ms · 29.7 K/s** | 60.8 ms · 16.1 K/s |
| **C** read-only                | **140 µs · 6.8 M/s** | 638 µs · 1530 K/s | 747 µs · 1307 K/s | 772 µs · 1265 K/s |
| **D** read-latest              | 54.7 ms · 17.9 K/s | 57.4 ms · 17.0 K/s | **46.0 ms · 21.3 K/s** | 57.3 ms · 17.0 K/s |
| **E** short-ranges (scan)      | **58.4 ms · 16.7 K/s** | 109.7 ms · 8.9 K/s | 69.5 ms · 14.0 K/s | 104.4 ms · 9.4 K/s |
| **F** read-modify-write        | 575 ms · 1739/s | 524 ms · 1907/s | **272 ms · 3.6 K/s** | 511 ms · 1956/s |

- **Pure-read C and scan-dominated E still go to UltimaDB** — reads never fsync,
  so its in-memory access path is untouched by the durability tier.
- **Write-heavy A/F and read-mostly B/D go to ReDB**, with UltimaDB mid-pack to
  last on the write-heavy two. Everything collapses to fsync latency: ~500
  fsyncs/burst pins all engines to single-digit-K/s, within ~2× of each other.
  UltimaDB's `Consistent` WAL fsync (plus per-commit snapshot promotion) is a
  touch heavier than ReDB's single-file page commit. **Durable per-op writes are
  not UltimaDB's strong suit; reads/scans are, regardless of tier.**
- Write-workload error bars are large (e.g. A/UltimaDB `554±75 ms`) — fsync
  latency is noisy on this shared host; treat the strict-tier ordering as
  directional, magnitudes as host-bound.

### Reproduce

```bash
cargo install critcmp                            # one-time
export CARGO_TARGET_DIR=/path/to/target          # if you use a shared target dir
export ULTIMA_BENCH_DIR=/path/on/real/disk       # REQUIRED on tmpfs hosts (e.g. /tmp = RAM)
for tier in nondurable strict; do
  ULTIMA_BENCH_DURABILITY=$tier cargo bench --bench ycsb_bench                       -- --save-baseline ultima_$tier
  ULTIMA_BENCH_DURABILITY=$tier cargo bench -p compare-benches --bench ycsb_fjall_bench   -- --save-baseline fjall_$tier
  ULTIMA_BENCH_DURABILITY=$tier cargo bench -p compare-benches --bench ycsb_rocksdb_bench -- --save-baseline rocksdb_$tier
  ULTIMA_BENCH_DURABILITY=$tier cargo bench -p compare-benches --bench ycsb_redb_bench    -- --save-baseline redb_$tier
done
critcmp -g '(.+)/[^/]+' ultima_nondurable fjall_nondurable rocksdb_nondurable redb_nondurable
critcmp -g '(.+)/[^/]+' ultima_strict     fjall_strict     rocksdb_strict     redb_strict
```

(`make bench/ycsb/compare` wraps the above. The competitor tier needs
`clang`/`libclang` for the RocksDB build.)
