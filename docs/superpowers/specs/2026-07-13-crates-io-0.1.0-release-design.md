# crates.io 0.1.0 release (ultima-db + ultima-vector) — design

**Date:** 2026-07-13
**Status:** Approved design (scope/tagging/auth decided by Peter)
**Context:** First public release. Both names verified unclaimed on crates.io
(sparse-index NoSuchKey for `ul/ti/ultima-db` and `ul/ti/ultima-vector`,
2026-07-13). All prerequisites landed: CI gate (ci.yml), sealed
`#[non_exhaustive]` builder API (task44), `repository`/`rust-version`
metadata, formal proofs matching code at T=64, published competitor bench
(2026-07-13), task51 bulk-append + the `bulk_load` corruption fixes.

## Decisions (fixed)

- **Scope:** publish BOTH `ultima-db` 0.1.0 and `ultima-vector` 0.1.0, in that
  order (vector's path dep gains `version = "0.1.0"`). Bench/infra crates
  (`bench_workloads`, `compare_benches`, `autobench`, `formal/kernel`) are
  never published; they keep no publish metadata.
- **Tagging:** annotated git tag `v0.1.0` on the release commit + GitHub
  release (`gh release create`) with the changelog text. Tag/release happen
  AFTER both crates are live on crates.io (so a publish failure can't strand
  a tag).
- **Auth/handoff:** Peter runs the actual `cargo publish` commands himself
  (`cargo login` first). Claude prepares everything up to and including
  dry-run/package verification, and resumes after publish for tag + release.

## Packaging — include whitelist (chosen approach)

`cargo package` for ultima-db currently ships 248 files (docs/ 116,
formal/ 30, bench-infra/ 20, tools/, .github/, rust-toolchain.toml…) because
the crate root is the repo root. Fix with an `include` whitelist per crate:

- **ultima-db** `Cargo.toml`:
  `include = ["src/**", "examples/**", "tests/**", "README.md", "LICENSE"]`
  (~40 files). Excluded by omission: docs/, formal/, bench-infra/, benches/,
  autobench/, scripts/, tools/, .github/, rust-toolchain.toml, Makefile,
  CLAUDE.md. **Two examples must ALSO be excluded** (audit 2026-07-13):
  `examples/profile_commit.rs` imports the unpublished
  `ultima_bench_workloads` (would break the published package) and
  `examples/fanout_microbench.rs` is an internal perf probe — exclude both
  via `exclude = ["examples/profile_commit.rs", "examples/fanout_microbench.rs"]`
  (cargo applies `exclude` on top of `include`) and verify their `[[example]]`
  declarations (if any) don't break packaging the same way `[[bench]]` might.
- **ultima-vector** `Cargo.toml`:
  `include = ["src/**", "examples/**", "tests/**", "README.md", "LICENSE"]`
  plus a new short `ultima_vector/README.md` (what it is, HNSW + SIMD
  highlights, quick-start, link to repo) and a copied
  `ultima_vector/LICENSE` (crates are packaged independently; the root
  LICENSE is not included in the vector package).

**The `[[bench]]` wrinkle (resolve empirically at execution):** the root
manifest declares 16 explicit `[[bench]]` targets (needed for
`harness = false`) whose files are excluded by the whitelist, and whose
dev-dep `ultima-bench-workloads` is path-only (auto-stripped at publish).

- Primary expectation: current cargo strips excluded-target sections from
  the packaged manifest (verify with `cargo package --list` + inspecting the
  generated `Cargo.toml` inside the `.crate`).
- **Fallback if cargo errors on explicit-but-excluded targets:** add
  `benches/**` to the include list and ship them. Publish verification and
  consumer `cargo build` / `cargo test` are unaffected (benches build only
  under `cargo bench` / `--all-targets`); only `cargo bench` on the
  extracted package would fail to resolve the stripped dev-dep — acceptable
  for 0.1.0, noted in CHANGELOG if taken.
- Whichever branch is taken, `tests/**` must be checked the same way: the
  packaged tests must compile with only versioned dev-deps (criterion, rand,
  serde, bincode, tempfile — all versioned; `ultima-bench-workloads` is the
  only path-only dev-dep and no `tests/` file may import it — verify by grep).

Rejected alternative: `exclude` list on the existing full package — ships
internal docs/proofs by default whenever a new top-level directory appears,
and is fragile in exactly the way that caused the 248-file package.

## Metadata polish

Both `[package]` sections gain:

- `keywords = ["database", "mvcc", "btree", "embedded", "snapshot"]`
  (ultima-db) / `["vector-search", "hnsw", "embedding", "simd", "database"]`
  (ultima-vector) — max 5, crates.io limit.
- `categories = ["database-implementations", "data-structures"]`
  (ultima-db) / `["database-implementations", "science"]` (ultima-vector;
  pick from the crates.io category slug list at execution — verify slugs
  exist, fall back to `database-implementations` alone).
- ultima-vector: `readme = "README.md"`, dep becomes
  `ultima-db = { path = "..", version = "0.1.0" }`.
- Both: `[package.metadata.docs.rs]` with
  `features = ["persistence", "fulltext", "metrics"]` (ultima-db) /
  `features = ["persistence"]` (ultima-vector) so docs.rs shows the full API.
  Verify locally with `cargo doc --no-deps --features …` (zero warnings not
  required; build success is the gate).

## Documentation completion workstream (folded in 2026-07-13, Peter's call)

Audit findings: 113 undocumented public items in ultima-db (concentrated in
`snapshot_stream` 44, `wal` 18, `metrics` 18, `error` 9, `fulltext` 8, plus
8 missing module docs; mostly struct fields/enum variants), 34 in
ultima-vector; only 3 doctests in all of ultima-db src; ultima-vector crate
doc is one line and it has no README. Core API (Store/Table/tx/config) is
already well documented; examples/ coverage is good and compiles on every
`cargo test`.

Requirements:

- Document ALL missing public items in both crates. Enforce permanently:
  add `#![warn(missing_docs)]` to both `src/lib.rs` roots — the existing
  clippy `-D warnings` CI gate then keeps coverage at 100%. (Verify clippy
  actually promotes rustc lints from the lib root under `-D warnings`; if
  not, use `#![deny(missing_docs)]` directly.)
- Doc quality bar: field/variant one-liners state meaning + units/invariants
  (not name-restating); module docs say what the module is FOR and name the
  entry type; error variants say when they occur and how callers typically
  react.
- Add runnable doctests to key entry points (target ≥8 across: `Store::new`
  + builder, `begin_write`/commit flow, `Table::define_index` + query,
  `insert_batch`, `Persistence::standalone_fast` (needs
  `persistence`-gated doctest), MultiWriter conflict-retry loop,
  `VectorCollection` insert+search, fulltext search). Doctests run under
  `cargo test` — they become permanent regression tests.
- ultima-vector: expand crate-level doc to a real intro (what/why/quick
  example, feature flags, dispatch/SIMD note) — this doubles as the bulk of
  its new README.
- README (ultima-db) additions beyond the currency pass: Installation
  section (`cargo add ultima-db`, feature-flag table), link to
  `docs/configuration.md` (GitHub URL), drop `ultima-journal` from the
  workspace table (separate repo — misleading on crates.io).

Keep-a-Changelog-ish format, one `## 0.1.0 — 2026-07-13` entry: initial

## Verification gates (in order, all must pass before handoff)

1. `make lint` + `cargo test --features persistence,fulltext,metrics` +
   `cargo test --lib` + `cargo test -p ultima-vector --features persistence`
   (the CI matrix, run locally on the release commit).
2. `cargo package --list -p ultima-db` and `-p ultima-vector`: inspect — no
   junk (docs/formal/bench-infra/.github), nothing missing (src, examples,
   tests, README, LICENSE), and the packaged manifests are sane.
3. `cargo publish --dry-run -p ultima-db` (full verify build of the package).
4. `cargo package -p ultima-vector` only — its dry-run cannot resolve a not-
   yet-published ultima-db; the real `cargo publish -p ultima-vector` after
   ultima-db is live serves as its verify. (If `--dry-run` happens to work
   via the path dep, run it too; do not treat its registry-resolution failure
   as a blocker.)
5. `cargo doc --no-deps` with the docs.rs feature sets, both crates.

## Publish handoff + post-publish (the exact sequence given to Peter)

```bash
cargo login                    # paste crates.io token (do this yourself)
cargo publish -p ultima-db
# wait for crates.io to index (usually <1 min; `cargo add ultima-db` in a
# scratch dir confirms), then:
cargo publish -p ultima-vector
```

After both are live, Claude: verifies both crates.io pages resolve, tags
`git tag -a v0.1.0 -m "UltimaDB 0.1.0 — first public release"` on the
release commit, pushes the tag, `gh release create v0.1.0` with the
changelog body, and checks docs.rs build status for both crates (async;
report, don't block).
