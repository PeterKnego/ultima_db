# crates.io 0.1.0 Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `ultima-db` 0.1.0 and `ultima-vector` 0.1.0 to crates.io with complete public-API docs, clean packages, a current README/CHANGELOG, and a tagged GitHub release.

**Architecture:** Three docs tasks (missing-docs completion gated by `#![warn(missing_docs)]`, doctests on key entry points, README/vector-README), then packaging metadata, then CHANGELOG + verification gates. The actual `cargo publish` is Peter's step (his token); tag + GitHub release land only after both crates are live. Spec: `docs/superpowers/specs/2026-07-13-crates-io-0.1.0-release-design.md`.

**Tech Stack:** cargo package/publish, rustdoc/doctests, gh CLI.

## Global Constraints

- Branch `feat/release-0.1.0` in an isolated worktree; merge to main when all tasks pass review; tag AFTER publish.
- `cargo clippy -- -D warnings` clean after every task (and `-p ultima-vector`); no `cargo fmt`.
- No API changes of any kind — docs, metadata, and packaging only. Any diff to a non-doc line of src code is out of scope (attributes like `#![warn(missing_docs)]` excepted).
- Versions stay 0.1.0 (both crates). MSRV stays 1.88.
- Doc quality bar (spec): field/variant docs state meaning + units/invariants, never restate the name; module docs say what the module is FOR and name its entry type; error-variant docs say when it occurs and how callers typically react.
- The CI matrix (`make lint`, featured tests, `--lib`, `-p ultima-vector`) must be green at every commit.

- [ ] **Step 1: Write `CHANGELOG.md`**

```markdown
# Changelog

## 0.1.0 — 2026-07-13

First public release of `ultima-db` and `ultima-vector`.

### ultima-db

- MVCC snapshot store on a persistent copy-on-write B-tree (T=64):
  zero-copy historical reads, O(1) table clones, atomic multi-table commits.
- Typed tables with auto-increment ids, unique / non-unique / custom
  secondary indexes (incl. an optional BM25 full-text index), and atomic
  batch operations; sequential batch inserts take an O(batch + height)
  bulk-append fast path.
- Concurrent writers (opt-in `MultiWriter`) with key-level OCC; opt-in
  serializable snapshot isolation (SSI) for write-skew prevention.
- Opt-in durability (`persistence`): group-committed WAL with Eventual /
  Consistent / ConsistentInline tiers, preallocated-WAL and coalesced write
  modes, CRC-protected checkpoints, crash recovery, checkpoint-only SMR
  mode, and the `Persistence::standalone_fast` preset.
- Bulk load / restore (O(N) sorted builds, multi-table atomic installs)
  and a streaming snapshot wire format.

### ultima-vector

- HNSW approximate-nearest-neighbor search over UltimaDB tables with
  metadata filtering and MVCC-consistent restores.
- SIMD distance kernels (Cosine / L2 / DotProduct / CosineNormalized) with
  runtime AVX-512 / AVX2 / NEON dispatch; strict input validation
  (dimension and non-finite checks) at every collection boundary.

### Notes

- MSRV 1.88. Pre-1.0 API: minor releases may contain breaking changes.
- Known limitations: SSI does not yet validate read-only transactions or
  index-DDL backfills (documented in `docs/tasks/task21`); CJK full-text
  tokenization is unigram-incomplete (task43); `BTree::from_sorted` packs
  one tail leaf below the MIN_KEYS floor at exactly
  `m·(MAX_KEYS+1)³ + δ (δ < MIN_KEYS)` element counts — benign and
  self-healing (task51).
```

Adjust the date if publish day differs. If Task 5 took the benches-included fallback, append that note here.
